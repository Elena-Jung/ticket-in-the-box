# ADR-0001: 하이브리드 클라우드 경계 설정

## Status

**Accepted** — 2026-04-15

## Context

이 프로젝트는 **가상 국내 리그 티켓팅 서비스**를 소재로 하는 DevOps/SRE
포트폴리오이다. 아키텍처 선택의 출발점은 **데이터의 물리적 위치와
컴퓨트의 분리 여부**이다.

다음 세 가지 현실 제약이 이 결정의 배경을 이룬다.

1. **CSAP 하등급 이슈** — AWS는 국내 CSAP 중 하(SaaS) 등급만 준수한다.
   공공/금융/일부 민감 도메인에서는 개인정보의 퍼블릭 클라우드 저장이
   규제 리스크를 내포하며, 티켓팅 도메인 또한 결제·본인 인증 정보를
   다루므로 이 논의에서 자유롭지 않다.

2. **개인 환경 제약** — 자택에 Rocky Linux 9.7 기반 SRV-1
   (Intel i5-6500, DDR4 8GB) 가용. 공인 IP 미할당,
   LG U+ 회선 + DDNS 환경.

3. **학원 AWS 계정** — 일정 수준 유료 사용 가능하나 영구성은 보장되지
   않음. IaC 기반 재현성이 필수 요구사항.

## Options Considered

### Option 1: All-in-AWS

모든 자원을 AWS 단일 리전에 구축.

- 장점: 아키텍처 단순, 네트워크 레이턴시 최소, 학습곡선 낮음
- 단점: 학원 계정 정책 변경 시 즉각적 비용/접근 리스크.
  CSAP 맥락의 내러티브 부재 → 포트폴리오 차별성 약함.

### Option 2: All-on-Premise

SRV-1 단일 노드에 K3s로 전체 스택 구성.

- 장점: 운영 비용 0원, 완전 독립, 외부 의존 없음
- 단점: 8GB RAM 제약으로 현실적 워크로드 구성 곤란.
  클라우드 운영 경험 부재 → DevOps/SRE 직무 적합성 저하.

### Option 3: Hybrid — 온프레 DB + AWS 컴퓨트 (채택)

SRV-1은 개인정보 원본 DB 전용, AWS는 애플리케이션과 관측 스택을 담당.
양자는 VPN 터널로 연결한다.

- 장점: 데이터 주권 내러티브 명확, 양쪽 운영 경험 획득,
  하이브리드 특유 장애 시나리오가 Phase 4 재료로 풍부함
- 단점: 네트워크 설계 복잡도 ↑, VPN 장애 시 서비스 영향,
  공인 IP 부재로 표준 Site-to-Site VPN 적용 불가

## Decision

**Option 3 (Hybrid) 채택.** 경계는 다음과 같이 설정한다.

### 온프레 (SRV-1)

- 개인정보 원본 DB (PostgreSQL)
- Tokenization 서비스 (원본 ↔ 토큰 변환 전담)
- 컬럼 단위 암호화 + 로컬 키 관리
- *상세: ADR-0002에서 후속 결정 예정*

### AWS

- EKS 기반 애플리케이션 (가상 티켓팅 서비스)
- 트랜잭션 DB (개인정보 없음, 토큰과 비즈니스 데이터만)
- 관측 스택 (Prometheus / Grafana / Loki)
- Ingress / ALB / Route53

### 연결

- WireGuard 기반 상시 VPN 터널 (SRV-1에서 AWS로 outbound)
- DDNS 변동에 대응하는 Persistent Keepalive 활용 예정
- *상세: ADR-0003에서 후속 결정 예정*

## Rationale

- **내러티브**: CSAP 맥락을 포트폴리오의 설득력 있는 근거로 활용.
  면접관이 "왜 하이브리드?"를 물었을 때 한 줄 답변 가능.
- **기술 범위**: 하이브리드가 아니면 경험하기 어려운 네트워크/보안/
  장애 시나리오를 포함 가능 → 차별화 포인트.
- **재현성**: 학원 계정 소멸 리스크는 Terraform 기반 IaC로 상쇄.
  재현 절차는 `scripts/reproduce.sh`(Phase 1에서 구현)로 담보.
- **선언형 철학과의 정합**: 이 경계 자체가 전체 설계의 Desired State가
  되며, Phase 1~4의 모든 작업은 이 경계를 구현·검증하는 과정이 된다.

## Consequences

### 장점

- Defense in Depth의 실제 구현 경험 (Layer 1: VPN, Layer 2: 데이터 분리)
- 하이브리드 특유 장애 시나리오 (VPN 단절, 온프레 지연, IP 변동 등)를
  Phase 4 Chaos Engineering 재료로 활용 가능
- 국내 현실에 기반한 내러티브 → 국내 기업 면접에서 공감대 형성 유리

### 감수할 트레이드오프

- 네트워크 레이턴시로 인해 p99 응답 시간 목표는 All-in-AWS 대비 보수적으로
  설정해야 함 (Phase 4에서 수치 목표 확정 예정).
- SRV-1 장애 시 서비스 전체 영향 → HA 설계를 별도 ADR에서 논의 필요.
- 초기 네트워크 구성 학습 곡선이 존재.

### 후속 작업

- **ADR-0002**: Data Segregation 전략 상세화
- **ADR-0003**: VPN 스택 선택 (WireGuard 유력, Phase 1 진입 전 확정)
- **ADR-0004**: Observability 스택 선택
- Phase 1 착수 시 상기 3개 ADR을 먼저 작성한다.
